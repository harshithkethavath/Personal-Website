What 60 Experiments Taught Me About Prompt Engineering Failure Modes

We ran a controlled study with 60 prompt variants on satellite cloud segmentation. Most failed in predictable ways. A few failed in ways that surprised us — and those surprises turned out to be the paper.

March 2026 8 min read EarthVision 2026 · CVPR Workshop

When we started this project, the question seemed simple: how much does prompt engineering matter for vision-language models doing satellite imagery segmentation? We had CLIPSeg, we had the CloudSEN12+ dataset, and we had a hypothesis that the right prompt could close a meaningful gap with supervised fine-tuning.

Sixty experiments later, the answer is: it matters, but not in the way we expected. And the way it doesn't matter turns out to be more interesting than the way it does.

The setup

We designed 60 prompt variants across several axes — specificity, technical language vs. plain language, task framing, spectral references, and combinations thereof. Each variant was evaluated on the same held-out CloudSEN12+ splits. We tracked IoU across cloud classes: thick cloud, thin cloud, cloud shadow, and clear sky.

The dataset is harder than it looks. Thin cloud and cloud shadow are spectrally ambiguous — thin cloud in particular can look like haze, like high-altitude cirrus, or like nothing at all depending on the underlying surface. This matters a lot for what came next.

What failed predictably

Most of the failure modes were unsurprising in retrospect. Overly generic prompts ("segment the clouds") performed worse than specific ones. Prompts that referenced spectral properties ("bright reflective regions") sometimes helped on thick cloud but confused the model on thin cloud. Technical jargon without grounding performed roughly the same as plain language.

The clearest finding was the 8-image threshold: below 8 labeled examples, sophisticated prompting was genuinely competitive with LoRA fine-tuning. Above it, supervised adaptation pulled ahead reliably. This crossover is stable across the prompt variants we tested — it's not sensitive to which prompt you use, it's sensitive to whether you have data.

The thing that surprised us

The interesting result was what happened to thin cloud and cloud shadow performance as we added more labeled data to fine-tuning. You'd expect monotonic improvement. What we saw was a dip — performance on these ambiguous classes temporarily degraded as we added labels, before recovering at higher data volumes.

The model was learning to be more confident in its representations before it had learned to be confident in the right ones. The ambiguous classes got worse before they got better.

We call this the class-level supervision dip. It's not a noise artifact — it replicates across seeds and across the prompt variants. And it has a practical implication: if you're fine-tuning for a spectrally ambiguous segmentation task and you evaluate at an intermediate data volume, you might conclude that fine-tuning isn't helping when it actually just hasn't converged on the hard classes yet.

What this means practically

A few takeaways I'd carry forward from this work. First, if you're data-limited (fewer than 8 examples), prompting is a legitimate strategy — don't dismiss it. Second, if you're fine-tuning for tasks with spectrally ambiguous classes, don't evaluate early and conclude it's not working. Third, the failure modes of prompting are fairly systematic and worth characterizing before writing it off.

The paper goes into more depth on the experimental design and the statistical analysis of the dip phenomenon. If you're working on similar problems, I'm happy to talk through the setup.
